# Tathastu Keepsakes — Pinterest Autopilot (Daily Auto-Publish)

**Date:** 2026-07-22
**Owner:** Akshat Garg (founder, Tathastu Keepsakes)
**Status:** Approved design — implementation in progress
**Mode:** Built autonomously per founder instruction ("take all decisions on your own, always
choose the recommended approach"). Recommended options are locked throughout.

---

## 1. Purpose & Scope

Build an agent that **autonomously publishes one Pinterest Pin every day** for Tathastu
Keepsakes, with zero human touch after setup. Each Pin:

- features a real product from the live catalog (`lib/products.json`, 54 products),
- links back to that product's page on `https://www.tathastukeepsakes.in`,
- carries **SEO-optimized metadata generated by Claude** — a catchy `title` (<100 chars),
  an engaging `description` (<500 chars, 3–4 natural related search terms, **no hashtags**),
  and a broad `board_suggestion` — matching the founder's original SEO-marketer spec,
- is posted via the **official Pinterest API v5** (ToS-clean, no scraping, no browser
  automation),
- runs on a **GitHub Actions cron** so it fires daily without a server.

### Explicitly OUT of scope
- No Instagram / WhatsApp automation (covered by the separate manual "Social Season" spec —
  see §8 reconciliation).
- No image generation or editing — we reuse the product photos already hosted on the site.
- No paid Pinterest ads, no analytics dashboards, no multi-account support.
- No video Pins or carousels in v1 (single image Pin only; carousel is a documented future
  extension).

### Success criteria
- After a one-time setup (Pinterest dev app + 4 GitHub secrets), the agent posts one Pin
  per day with no human involvement.
- Metadata obeys the SEO contract exactly (enforced by a Zod schema at decode time).
- No product repeats until the catalog is exhausted (least-recently-posted rotation).
- A run never double-posts, and a failed run leaves state clean so the next day recovers.
- Secrets are never committed; everything sensitive lives in encrypted GitHub secrets.

## 3. Architecture — Modules

All code under `scripts/pinterest/`. Each module has one job, a typed interface, and is
unit-testable in isolation. Pure logic (selection, URL building, validation) is separated
from I/O (network, filesystem) so tests need no network.

```
scripts/pinterest/
├── config.ts       Env loading + validation (fail fast with a clear message)
├── types.ts        Shared types: Product, SeoContent, PostedLog, PinResult
├── products.ts     loadProducts(), selectProduct() rotation, buildPublicImageUrl()
├── seo.ts          generateSeo() — Claude + zodOutputFormat, SEO contract enforced
├── auth.ts         getAccessToken() — refresh-token → fresh access token per run
├── pinterest.ts    resolveBoard() (get-or-create), assertImageReachable(), publishPin()
├── state.ts        readLog(), recordResult() — posted-log.json read/append
├── run.ts          orchestrator: wire everything, one Pin per invocation
└── state/
    └── posted-log.json   committed de-dupe + rotation store
```

**Data flow (one daily run):**

```
run.ts
  1. config.load()                       → validated env (fails fast if a secret is missing)
  2. state.readLog()                     → history of what's been posted
  3. products.selectProduct(all, log)    → least-recently-posted product (pure)
  4. seo.generateSeo(product)            → {title, description, board_suggestion} (Claude+Zod)
  5. auth.getAccessToken()               → fresh Pinterest access token (refresh flow)
  6. pinterest.assertImageReachable(url) → HEAD/GET precheck; skip+log if 404
  7. pinterest.resolveBoard(board_sug)   → board_id (get-or-create by name)
  8. pinterest.publishPin(...)           → POST /v5/pins → pin id
  9. state.recordResult(...)             → append to posted-log.json (committed by CI)
```

Each step logs a structured line. Any thrown error fails the CI job (visible in the Actions
tab) **without** corrupting state — the log is only appended after a confirmed publish.

## 4. Component Contracts

### 4a. `products.ts`
- `loadProducts(): Product[]` — reads `lib/products.json`, coerces to the fields we need
  (`id`, `name`, `description`, `images`, `category`, `price`, `originalPrice`).
- `buildPublicImageUrl(relativePath): string` — prefixes `SITE_ORIGIN`, guarantees a raw
  public URL Pinterest can fetch (never the `/_next/image` form).
- `selectProduct(products, log): Product` — **least-recently-posted** rotation: pick the
  product whose most-recent successful post is oldest (never-posted products sort first, by
  catalog order for determinism). Pure function, fully unit-tested.

### 4b. `seo.ts`
- `SeoSchema` (Zod): `title` (≤100), `description` (≤500, **regex-rejects `#`**),
  `board_suggestion` (short, 2–4 words).
- `generateSeo(product): Promise<SeoContent>` — one `client.messages.parse()` call to
  `claude-opus-4-8` with `output_config: { format: zodOutputFormat(SeoSchema) }`. System
  prompt encodes the expert-Pinterest-SEO-marketer persona + hard rules (hook + primary
  keyword in title; 3–4 natural related search terms in description; no hashtags; Indian
  gifting market). Reads `message.parsed_output`. No `thinking` (constrained decoding is the
  hard requirement; bounded copywriting task).

### 4c. `auth.ts`
- `getAccessToken(): Promise<string>` — `POST https://api.pinterest.com/v5/oauth/token`
  with `Authorization: Basic base64(APP_ID:APP_SECRET)`,
  `Content-Type: application/x-www-form-urlencoded`,
  body `grant_type=refresh_token&refresh_token=<PINTEREST_REFRESH_TOKEN>`. Returns fresh
  `access_token`. This is what makes true daily autonomy possible (access tokens expire
  ~30 days; the refresh token is long-lived).

### 4d. `pinterest.ts`
- `assertImageReachable(url): Promise<void>` — GET/HEAD precheck; throws a clear "image not
  reachable" error so we never send Pinterest a URL it can't fetch.
- `resolveBoard(token, name): Promise<string>` — `GET /v5/boards` (paginate via `bookmark`),
  case-insensitive name match → return `id`; else `POST /v5/boards` `{name, privacy:"PUBLIC"}`
  → return new `id`.
- `publishPin(token, {board_id, title, description, link, alt_text, image_url}): Promise<Pin>`
  — `POST /v5/pins` with `media_source:{source_type:"image_url", url}`. Returns the created
  Pin (id).

### 4e. `state.ts`
- `posted-log.json` shape:
  `{ "version": 1, "posts": [ { "productId", "pinId", "boardId", "boardName", "title", "link", "postedAt" } ] }`.
- `readLog()` tolerates a missing/empty file (returns an empty log).
- `recordResult(entry)` appends and writes back (pretty-printed for clean diffs). CI commits
  the change so the next run sees updated history.

### 4f. `run.ts`
- Orchestrates the 9-step flow. `--dry-run` flag: do everything **except** the live
  `publishPin` + `recordResult` (prints the payload) — used for local/CI smoke tests without
  posting. Exit non-zero on any hard failure.

## 5. Scheduling & Secrets (GitHub Actions)

- **Workflow:** `.github/workflows/pinterest-daily.yml`.
- **Trigger:** `schedule: cron '30 3 * * *'` (03:30 UTC = **09:00 IST**) + `workflow_dispatch`
  (manual "run now" button, supports a dry-run input).
- **Steps:** checkout → setup-node 22 (npm cache) → `npm ci` → `npx tsx scripts/pinterest/run.ts`
  → commit `posted-log.json` back (only if changed) using the built-in `GITHUB_TOKEN` with
  `permissions: contents: write`.
- **Encrypted secrets (never committed):**
  `ANTHROPIC_API_KEY`, `PINTEREST_APP_ID`, `PINTEREST_APP_SECRET`, `PINTEREST_REFRESH_TOKEN`.
- **Non-secret config** (env in the workflow): `SITE_ORIGIN=https://www.tathastukeepsakes.in`.
- **Failure behavior:** any error → red X in the Actions tab; state uncorrupted; next day
  retries. (A notification hook is a documented future extension.)

## 6. Learning-Mode Contribution Hooks

Two decisions genuinely shape behavior and are the founder's to tune (sensible defaults
shipped so the system runs unattended tonight):

1. **`selectProduct()` rotation policy** (`products.ts`) — default is least-recently-posted.
   Alternative strategies (weighted by price band, category round-robin, seasonal boosting)
   are viable; the signature + default are in place with a `★ Insight` note.
2. **SEO system prompt** (`seo.ts`) — the persona/voice/keyword rules. Default encodes the
   expert-Pinterest-SEO-marketer brief; the founder can tune tone, keyword aggressiveness,
   or CTA style. Marked with a `★ Insight` note.

## 7. Testing

- **Unit (Jest, no network):** `selectProduct` rotation correctness (never-posted first,
  oldest-first, deterministic tie-break); `buildPublicImageUrl` (relative → public, rejects
  optimizer form); `SeoSchema` validation (accepts valid, rejects >100 title, >500 desc, any
  `#`); `state` append/dedupe + tolerant read of missing file.
- **Integration smoke:** `run.ts --dry-run` executes selection + SEO + payload assembly
  without posting (guarded so CI never posts).
- **Typecheck:** `npx tsc --noEmit` (matches existing CI); strict mode.

## 9. Assumptions & Risks

- **Assumption:** product images remain publicly reachable at the derived URLs. Mitigation:
  `assertImageReachable` precheck skips + logs instead of sending a bad URL.
- **Assumption:** the Pinterest app stays approved for standard access and the refresh token
  stays valid. Risk: if Pinterest revokes/rotates it, runs 401. Mitigation: clear auth-error
  message in logs; SETUP.md documents re-minting the refresh token.
- **Risk:** catalog drift (price/SKU changes). Low impact — we read `products.json` fresh
  each run, so new products auto-enter rotation and removed ones drop out.
- **Risk:** Claude returns copy that's on-contract but off-brand. Mitigation: Zod guarantees
  shape/limits; the founder can tune the system prompt (hook #2).
- **Risk:** two runs in one day (manual dispatch + cron). Mitigation: idempotency is
  best-effort via the log; documented that manual dispatch posts an additional Pin by design.

## 10. What Success Looks Like (post-implementation)

The founder sets up the Pinterest app and 4 GitHub secrets once. Every morning at 09:00 IST,
GitHub Actions selects the next product, has Claude write SEO-perfect metadata, publishes the
Pin to the right board via the official API, and commits the updated log. Pinterest stays fed
daily, on-brand and on-SEO, with no ongoing effort.
